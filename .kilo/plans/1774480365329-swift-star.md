# Refactoring Plan: Project Structure Improvement

## Objective
Refactor the project structure to better organize models, schemas, and client code by:
1. Creating a `models/` directory with subdirectories for admin, courier, and customer entities
2. Moving shared models to the models root
3. Creating a `utils/clients/` directory for external service clients
4. Making minimal code changes (import updates only) without altering business logic

## Current Structure Analysis
- `src/models.py`: Contains all SQLAlchemy models (502 lines)
- `src/schemas.py`: Contains all Pydantic schemas (458 lines)
- `src/utils/`: Contains utility modules but lacks client organization
- `src/paylink_client.py`: Payment gateway client in root
- `src/utils/sms.py`: SMS client already in utils

## Detailed Migration Steps

### Phase 1: Create Directory Structure
1. Create `src/models/` directory and subdirectories
2. Create `src/schemas/` directory and subdirectories  
3. Create `src/utils/clients/` directory
4. Add `__init__.py` files to all new directories

### Phase 2: Migrate Models
1. Move `Admin` class to `src/models/admin/admin.py`
2. Move `CourierProfile` class to `src/models/courier/courier_profile.py`
3. Move `CustomerProfile` class to `src/models/customer/customer_profile.py`
4. Distribute remaining models to appropriate files in `src/models/` root:
   - `src/models/user.py`: User model
   - `src/models/order.py`: Order model
   - `src/models/invoice.py`: Invoice model
   - `src/models/conversation.py`: Conversation model
   - `src/models/message.py`: Message model
   - `src/models/wallet.py`: Wallet model
   - `src/models/payment.py`: Payment model
   - `src/models/promocode.py`: Promocode model
   - `src/models/city.py`: City model
   - `src/models/important_event.py`: ImportantEvent model
   - `src/models/courier_review.py`: CourierReview model
   - `src/models/refresh_token.py`: RefreshToken model
   - `src/models/deposit_request.py`: DepositRequest model
   - `src/models/coupon_balance_addition.py`: CourierBalanceAddition model
   - `src/models/promocode_usage.py`: PromocodeUsage model
5. Update imports in each migrated file to reference new locations
6. Update `src/models/__init__.py` to export all models for backward compatibility

### Phase 3: Migrate Schemas
1. Analyze schemas.py to determine categorization
2. Move admin-related schemas to `src/schemas/admin/admin.py`
3. Move courier-related schemas to `src/schemas/courier/courier.py`
4. Move customer-related schemas to `src/schemas/customer/customer.py`
5. Move shared schemas to `src/schemas/shared/` directory
6. Update `src/schemas/__init__.py` to export all schemas for backward compatibility

### Phase 4: Migrate Clients
1. Move `src/paylink_client.py` to `src/utils/clients/paylink.py`
2. Move `src/utils/sms.py` to `src/utils/clients/sms.py`
3. Update imports in files that use these clients

### Phase 5: Update Imports
1. Update all imports throughout the codebase to reference new locations
2. Focus on:
   - Router files importing models/schemas
   - Utility files importing models/schemas
   - Main application files
   - Any other files referencing moved code
3. Use absolute imports from `src` root for clarity

### Phase 6: Verify and Test
1. Run import checks to ensure no broken references
2. Run existing tests to verify functionality unchanged
3. Verify that all models and schemas are accessible through their new paths

## Backward Compatibility Strategy
To minimize disruption:
1. Maintain temporary backward compatibility by keeping original files during transition
2. Update imports gradually, verifying at each step
3. Remove original files only after confirming all imports work correctly
4. Use import aliases where needed to maintain compatibility during transition

## Estimated Effort
- Directory creation: 15 minutes
- Model migration: 60-90 minutes
- Schema migration: 45-60 minutes  
- Client migration: 15 minutes
- Import updates: 90-120 minutes
- Testing and verification: 30-60 minutes
- Total: 4-5 hours

## Risk Mitigation
- No business logic changes - only file relocation and import updates
- Incremental approach allows for rollback at any step
- Comprehensive testing after each phase
- Backup of original files before migration
